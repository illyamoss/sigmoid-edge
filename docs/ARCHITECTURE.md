# TARGET ARCHITECTURE BLUEPRINT & SYSTEM BOUNDARIES

This document outlines the strict architectural rules, directory structures, and data flow patterns for this application. Every file generated by the engineering agent must strictly comply with these rules. 

---

## 1. DIRECTORY STRUCTURE

The application enforces strict Clean Architecture principles adapted for Next.js (App Router) and React 19. The codebase is divided into clear horizontal layers and highly isolated vertical feature modules.

```text
src/
├── app/                  # ROUTING LAYER (Framework Entry)
│   ├── layout.tsx        # HTML wrapper, global themes, root context providers.
│   ├── page.tsx          # Public-facing root landing page.
│   ├── api/              # Standard Next.js server Route Handlers (e.g., Stripe webhooks).
│   └── (dashboard)/      # Authenticated application route group.
│
├── components/           # PRESENTATION LAYER (Pure UI Layout)
│   ├── ui/               # Abstract Atomic Primitives (shadcn/ui - Tailwind, Radix).
│   ├── shared/           # Cross-cutting application layouts (Navbar, Sidebar, Modals).
│   └── features/         # Highly isolated, domain-specific component modules.
│
├── core/                 # BUSINESS LOGIC LAYER (Framework Agnostic, Pure TS)
│   ├── domain/           # Entities, Enums, Type declarations, and Zod Schemas.
│   ├── use-cases/        # Orchestrators of pure business rules and logic.
│   └── services/         # Service abstract interfaces and infrastructure contracts.
│
├── lib/                  # INFRASTRUCTURE LAYER (Concrete Implementations)
│   ├── db/               # Database ORM clients and connection initializers.
│   └── integration/      # Real implementations of core service contracts (Stripe, Clerk).
│
└── hooks/                # CLIENT-SIDE LAYER (Custom UI/State triggers)
    ├── use-toast.ts      # Abstracted component notification triggers.
    └── use-media-query.ts# Global responsive screen width listener hooks.
```

## 2. STRICT ARCHITECTURAL BOUNDARIES

### 2.1 The Dependency Inversion Rule
* Dependencies must always flow **inward**. 
* The `core` layer contains pure business logic. It must **never** import from `app`, `components`, `lib`, or `hooks`.
* Infrastructure code inside `lib/` must fulfill and implement the strict interfaces declared inside `core/services/`.

### 2.2 Server Components (RSC) vs Client Components
* Every component is a React Server Component (RSC) by default.
* Data fetching and initial page layout hydration must happen natively on the server (`page.tsx` level).
* The `'use client'` directive is restricted exclusively to low-level layout nodes (leaf nodes) that depend on event listeners, browser DOM APIs, or React client state hooks (`useState`, `useActionState`).

### 2.3 Strict Data Flow & Mutation Patterns
1. **Reads**: Page (Server Component) -> Invoke Core Use Case -> Query DB via Lib client -> Render View.
2. **Writes/Mutations**: Component View -> Trigger secure Next.js Server Action -> Validate data schema natively via Zod (`core/domain/`) -> Process business rule inside Use Case -> Commit changes to Lib infrastructure -> Return clean JSON payload back to view.

## 3. UI SWAPPABILITY & ABSTRACTION STRATEGY

### 3.1 Design System Isolation
* Feature-level UI files inside `src/components/features/` must **never** import Radix UI primitives or third-party style packages directly. 
* All styling abstractions must pass exclusively through the atomic wrappers inside `src/components/ui/` (the `shadcn/ui` layer). If a button package changes, it is modified inside `src/components/ui/button.tsx` without leaking side effects into any app business logic.

### 3.2 Dynamic Component Composition
* Multi-variant layout files must accept child layouts or variable components using explicit typescript functional props.
* Avoid absolute, tight coupling between feature components. Pass contextual configuration down from the core rendering layers to ensure feature modules remain highly reusable and easily interchangeable.

### 3.3 Micro-Interactions & Animation Standards
* All animation structures must rely on `framer-motion`.
* Complex hover effects, structural canvas shifting, or layout animations must be isolated in leaf-level client blocks to safeguard primary page loading speeds.
